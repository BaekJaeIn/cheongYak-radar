## P1. 지원 — 신혼부부 청약 준비자 (Primary User)

| 항목 | 내용 |
|---|---|
| **역할** | 앱의 주 사용자 (본인/배우자) |
| **상황** | 경기 안양(평촌)·군포(산본) 거주 희망, 신혼부부 특별공급 관심 |
| **기술 수준** | 일반 스마트폰 사용자, 모바일 우선 |
| **목표** | 관심 지역의 신규 청약·임대 공고를 놓치지 않고, 내 조건(면적·순위·신혼부부)에 맞는지 빠르게 판단 |
| **불편(Pain)** | 청약홈·마이홈·SH 등 사이트가 분산돼 매일 확인이 번거롭고, 자격조건 원문이 어려움 |
| **동기(Motivation)** | 마감 전에 적합 공고를 발견하고, 자격 여부를 쉽게 이해하고 싶음 |
| **주요 행동** | 매일 아침 신규 공고 확인 · 필터 설정 · 상세 검토 · 북마크 · 알림 수신 |

**핵심 니즈**
- 오늘 신규 공고를 한눈에 (NEW 배지, D-day)
- 관심 지역/면적/유형/순위/신혼부부 필터
- 자격조건을 쉬운 말로 요약("나한테 해당되나요?")
- 관심 공고 저장 + 마감 임박 알림

## P2. 시스템 — 데이터 파이프라인 운영자 (System / Operator Perspective)

| 항목 | 내용 |
|---|---|
| **역할** | 자동수집 파이프라인의 운영 관점 (실질적으로 본인이 겸하지만, 시스템 신뢰성 요구를 대변) |
| **상황** | Supabase Edge Function cron이 매일 07:00 4개 외부 소스에서 공고 수집 |
| **기술 수준** | 개발자/운영자 |
| **목표** | 외부 API/크롤링이 일부 실패해도 안정적으로 최신 공고를 `notices`에 정합성 있게 유지 |
| **불편(Pain)** | 외부 API 불안정, SH 크롤링 구조 변경, API 키 미발급(초기엔 목업 필요) |
| **동기(Motivation)** | 손이 가지 않는 자동수집, 중복 없는 데이터, 키 노출 없는 안전한 구성 |
| **주요 행동** | cron 수집 · upsert · 소스별 에러 격리 · 목업/실데이터 전환 · 키를 서버에만 보관 |

**핵심 니즈**
- 소스별 독립 수집 + 에러 격리 (NFR-4)
- 공고번호 기준 upsert 정합 (NFR-5)
- API 키 확보 전 목업 데이터로 개발 (C-1)
- 비밀정보 서버 보관 (NFR-3)
